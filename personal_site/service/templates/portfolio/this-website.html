Design

The goal architecture for this websites backend was something akin to this structure. The concept here is you have Load Balancer exposed to the internet that then talks to a more secure internal network where nothing actually accesses the open net.
This is nice for a few reasons.

  • Secure. Having a Load Balancer as the gateway for your network lets you keep your servers more secure
  • Scalable. You can easily add more projects to this backend
  • Versatile. Whether your backend service is a VM, Lambda Function, Kubernetes Cluester, or a plumbus this architecture can support it

Cost 🤑

There's one problem with this. There is no way to do this for free. See, all 3 of the big cloud providers (AWS, GCP, Azure) provide free tiers. Some are only free for 12-months, some are free for life. I encourage you to read through the quota in the free tier and maximize these before you pay anything to a cloud provider. However, key for the above setup, no one gives you a load balancer for free.
So are we shit ouf of luck? No, we just have to work a little bit harder, and learn some basics about how web security works Yay!

Step one, we replace our load balancer with a vm that runs NGINX (We will get into what this is and how to set it up soon), which means our architecture looks like this now

Why is havimg a VM important?

To be fair, NGINX is the important part for fixing our missing load balancer issue. You could host it on anything, but most SASS solutions don't let you use custom dockerfile for cheap. For example, in GCP's free tier you get a pretty good amount of access to app-engine, but this is only for standard containers. This means the containers have to be from a set of premade ones google provides. This is still nice for doing something like hosting a Flask/Node/SpringBoot application, but not NGINX.
Using a vm allows for you to install docker in the vm and run a custom container with NGINX inside of it. You could also run NGINX directly on the VM, but I think thats bad practice.

DNS Setup

Great, we have a website up (at the very least nginx proxy manager on port 81), but my only way to access is the external ip address, how do I get a real url?
First things first, buy a url. I recommend using Namecheap for this. You might notice that .com URLs are expensive, hence why this is a .club lol. There are plenty of fun TLD (Top Level Domains), but I wouldn't go too crazy. Ultimately someone still runs that TLD so maybe buying a .cum url is funny, but how much do you trust that guy? Objectively me owning a .club url is probably dumb (look at this sketchy ass wikipedia page lol https://en.wikipedia.org/wiki/.club). Realistically you're fine, but try to stick to some of the more well known cheap TLD like .xyz, .gg, etc. If you can snag a .com or .net for cheap because your url is cheap than awesome!
One last point on this topic, name cheap has an ai url generator and its terrible in a great way. 10/10 you should have fun with it https://www.namecheap.com/domains/domain-name-generator/
Anyway, you bought a url. Great. Fantastic. Now what? Well you need to setup the DNS. In name cheap you can use the "Advanced DNS" to set this up. Point an A record to your NGINX servers external IP address and then make a CNAME for any subdomains ( subdomain.url.com, the most common is www obviously. You can lookup other standard subdomains if you're interested in the topic ) Even with just the @ record setup you can now go to your-custom-url.com:81 and setup the nginx proxu manager

nginx, a reverse proxy.
We assign the free external ip address GCP allows to the free tier vm. On this vm setup Nginx. I recommend using open source software Nginx Proxy Manager This supplies a ui to simplify setup for beginners and has a built in connection to letsencrpyt to set you up with SSL (and therefore HTTPS) with a button click! I recommend a custom nginx setup if you're willing, but proxy manager makes the task more approachable and I use it myself for now because I'm lazy :).
